# Documentation Philosophy

Documentation exists to transfer deep operational understanding from the implementation into the mind of the reader.

The goal is not to describe files or code mechanically.

The goal is to teach:
- how the system works,
- why it works that way,
- how components interact,
- what assumptions exist,
- what behavior emerges from the implementation,
- and what consequences changes would have.

The reader should finish the document with a strong mental model of the system and be able to reason about behavior without opening source files.

Documentation must feel like it was written by an engineer who deeply understands both the implementation and the operational realities of the system.

# Core Principles

## Single Source of Truth

Each document is the definitive reference for its topic.
Do not duplicate explanatory content across files.

## Explain Meaning, Not Just Mechanics

Do not merely describe steps or functions.

Explain:
- intent,
- consequences,
- relationships,
- tradeoffs,
- guarantees,
- risks,
- and operational significance.

Every major mechanism must answer:
- What does this actually accomplish?
- Why does the system need this?
- What problem does it prevent?
- What would happen if it behaved differently?
- How does it affect surrounding systems?

## Systems Thinking First

Always document systems as interacting behaviors rather than isolated code sections.

Explain:
- data flow,
- control flow,
- lifecycle flow,
- dependency relationships,
- state propagation,
- synchronization,
- and cascading effects.

## Reader-Oriented Technical Writing

Assume the reader is intelligent and technical but unfamiliar with the implementation.

Optimize for:
- comprehension,
- clarity,
- operational understanding,
- and long-term maintainability knowledge.

Avoid implementation narration.

Instead of:
"This method checks if X exists."

Prefer:
"This validation prevents the system from entering an unrecoverable state where downstream processing would operate on incomplete transactional context."

## Honest and Precise

Never exaggerate correctness, safety, or guarantees.

Document:
- limitations,
- edge cases,
- failure behavior,
- implicit assumptions,
- and fragile areas honestly.

## Explain Dynamic Behavior

Static descriptions are insufficient.

Focus heavily on:
- runtime behavior,
- transitions,
- reactions,
- retries,
- recovery,
- sequencing,
- timing,
- and coordination.

# Required Content Depth

Every document must thoroughly explain:

- Architectural purpose
- Core concepts and mental models
- Component responsibilities
- State machines and transitions
- Runtime lifecycle behavior
- Configuration effects
- Failure and recovery behavior
- Concurrency and synchronization implications
- Performance characteristics
- Scalability implications
- Resource management
- Operational risks
- Non-obvious invariants
- Cross-component interactions
- Edge cases and degraded behavior
- Important implementation quirks
- Safety guarantees and limitations

# Writing Style

## Tone

Write like a senior systems engineer explaining a production system to another senior engineer.

Professional, calm, precise, deeply informed.

## Structure

Use:
- `##` for primary sections
- `###` only when genuinely necessary

Prefer flowing prose over fragmented bullets unless enumeration improves clarity.

## Language

Use direct and precise language.

Avoid:
- filler,
- repetition,
- generic explanations,
- marketing tone,
- and low-information statements.

Every paragraph must contribute meaningful understanding.

## Technical Depth

Always prioritize:
1. understanding,
2. causality,
3. operational implications,
4. architectural reasoning.

Over:
- implementation trivia,
- syntax explanation,
- or file-by-file reporting.

## Formatting

Use backticks for:
- identifiers,
- statuses,
- config keys,
- protocol values,
- exact runtime states.

Humanize machine-oriented values when useful.

Example:
`TX_PENDING` should be explained as “Transaction Pending”.

# Documentation Quality Standard

A high-quality document should allow a future engineer to:
- safely modify the system,
- reason about runtime behavior,
- predict side effects,
- diagnose production issues,
- understand architectural intent,
- and identify operational risks,

without needing to reverse-engineer the implementation again.
