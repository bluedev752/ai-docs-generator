# Documentation Style Guide

This document defines the exact standard and process for writing every `.md` file. Every future documentation file must match the defined depth, tone, structure, clarity, and quality.

## Core Principles

- **Single source of truth**: Each documentation file is the definitive, complete reference for its topic. No other file may duplicate its explanatory content.

- **Zen, minimal, clean, elegant**: Use precise language. Eliminate redundancy, repetition, filler, and unnecessary words. Every sentence must add unique value.

- **Thorough and honest**: Document every concept, flow, trigger, state transition, edge case, quirk, invariant, and implication exactly as implemented. Never oversimplify or omit important details.

- **Professional tone**: Write for an experienced engineer or project owner who needs to understand the system deeply without reading source code.

- **Human-readable yet technical**: Use clear prose. Avoid code snippets unless essential for understanding. Prefer explaining behavior, intent, and system relationships over reproducing implementation details.

- **Explain systems, not just files**: Documentation must explain how components influence one another, why certain flows exist, what assumptions are relied upon, and how changes propagate through the system.

- **Teach the reader the mental model**: The goal is not only to describe what the code does, but to transfer deep operational understanding to the reader.

## Required Structure

Every documentation file must contain at least four top-level sections.

Recommended top-level sections (adapt as needed for the topic):

1. Introduction / Overview
2. Core Concepts / Key Components
3. Detailed Behavior / Processes / Flows
4. Configuration
5. Operational Considerations / Edge Cases
6. Security, Reliability, and Scalability Implications
7. Important Notes

The structure should feel natural and balanced. Sections should evolve from high-level understanding into progressively deeper operational detail.

## Content Requirements

- Explain both the **how** and the **why** of every important mechanism.
- Explain how components interact with and affect one another.
- Clarify the intent behind architectural decisions when inferable from implementation.
- Describe all state machines, status values, and transitions with their triggers and outcomes.
- Document configuration keys, defaults, fallbacks, and the exact behavioral effect of each value.
- Cover error handling, failure recovery, retries, fallback behavior, and degraded operation modes.
- Explicitly describe behavior when features are disabled, partially configured, unavailable, or misconfigured.
- Include operational implications, performance considerations, concurrency concerns, atomicity guarantees, and resource management behavior where relevant.
- Document lifecycle behavior, initialization flows, cleanup behavior, caching behavior, synchronization behavior, and persistence implications where applicable.
- Explain not only what happens, but what that means for operators, developers, integrators, or maintainers.
- Include a dedicated "Important Notes" section capturing non-obvious invariants, gotchas, assumptions, and implementation-sensitive behavior.
- Surface hidden coupling and implicit dependencies whenever they materially affect behavior.
- End with a closing statement declaring the file as the single source of truth for its topic.

## Depth Expectations

The documentation must go beyond surface-level narration.

Do not merely describe the order of operations. Explain:
- why the order matters,
- what assumptions exist between steps,
- what guarantees are established,
- what side effects occur,
- and how failures or state changes propagate through the system.

The reader should come away understanding:
- how the system behaves,
- why it behaves that way,
- how different parts influence one another,
- and what consequences emerge from changing behavior or configuration.

Avoid "read-and-report" style documentation.

Instead, produce documentation that demonstrates genuine comprehension of the system.

## Style Rules

Use `##` for main sections and `###` for subsections only when genuinely required.

- Use backticks for exact identifiers.
- Status values and standardized strings should be formatted for readability. For example, `TX Pending` instead of `"tx_pending"` when appropriate.
- Never start sections with code or tables.
- Use bold for key terms only when it meaningfully improves clarity.
- Prefer dense, high-value prose over fragmented micro-paragraphs.
- Write in present tense.
- Use active voice.
- Use bullet lists for enumerations, never for primary explanatory prose.
- Avoid vague wording such as "basically", "simply", "just", or "somehow".
- Do not add "TODO", "FIXME", speculation, assumptions without basis, or hypothetical behavior not grounded in implementation.
- Do not reference line numbers or internal function locations unless part of the public API surface.

## Process for Creating New Documentation

1. Read the corresponding source file(s) completely and multiple times.
2. Read all directly related files including callers, consumers, configuration, tests, integrations, and state definitions.
3. Build a complete mental model of the feature and its operational behavior.
4. Map every concept, interaction, dependency, edge case, and failure path.
5. Write the documentation following this guide exactly.
6. Review against the source to eliminate inaccuracies, omissions, shallow explanations, or misleading simplifications.
7. Ensure the document can stand entirely on its own as the authoritative reference for the topic.
8. Ensure the document does not go into detail about stuff that is off-topic and not its responsibility.


Every new file must reach or exceed that standard.
